### 1.2 现有 MVP 资产规模

- **页面组件**: 50+ 个独立页面组件
- **UI 基元**: 40+ shadcn/ui + Radix UI 组件
- **翻译键**: 500+ 中英双语翻译
- **测试**: 单元测试(Vitest) + E2E 测试(Playwright)
- **主题**: Cyberpunk 霓虹 + Liquid Glass 液态玻璃双主题
- **AI 集成**: 支持 8+ 模型提供商(OpenAI/Claude/DeepSeek/Qwen 等)
- **状态管理**: Zustand + React Context 多层架构

### 1.3 现有架构优势

- **单元自治**: 每个模块独立 Context/Store，独立的生命周期和测试
- **Provider 层级**: ErrorBoundary → Theme → I18n → App → Contacts → AI Model
- **双模式**: Standalone（全屏桌面）+ Widget（浮动浮窗）
- **无障碍**: 基于 Radix UI 原语，支持 WCAG 标准

### 1.4 当前存在的不足与改进空间

| 维度           | 当前不足                                            | 影响                   |
| -------------- | --------------------------------------------------- | ---------------------- |
| **数据持久化** | 大部分数据基于内存模拟/LocalStorage，缺少后端持久化 | 无法实现跨会话数据共享 |
| **用户认证**   | 无用户登录/注册/权限管理                            | 多用户场景不可用       |
| **多租户**     | 单租户架构                                          | 无法支撑 SaaS 多组织   |
| **后端 API**   | 无正式后端服务，AI API 直接从前端调用               | 安全性差，无法扩展     |
| **实时协作**   | 仅有基本的协同创作，缺少实时同步                    | 团队协作受限           |
| **数据分析**   | 已有可视化但缺少深度分析引擎                        | 决策支持不充分         |
| **自动化**     | 手动操作居多，自动化流程有限                        | 运营效率受限           |
| **移动端适配** | 主要针对桌面端，移动端体验有待优化                  | 覆盖场景有限           |
| **插件系统**   | 无扩展机制                                          | 第三方集成困难         |

## 6. 可扩展性设计原则

### 6.1 架构原则

1. **模块化**: 每个功能作为独立模块，通过接口通信
2. **分层清晰**: API 网关 → 服务层 → 数据访问层
3. **事件驱动**: 采用 Event Bus 处理异步流程
4. **服务化**: 核心能力抽象为微服务
5. **数据隔离**: Row-Level Security 保证多租户安全

### 6.2 技术选型

| 层面     | 技术                    | 理由                                |
| -------- | ----------------------- | ----------------------------------- |
| 后端框架 | NestJS                  | TypeScript 全栈，装饰器模式，模块化 |
| 数据库   | PostgreSQL + Redis      | 关系型 + 缓存 + 消息队列            |
| ORM      | Prisma                  | 类型安全，迁移管理                  |
| API      | REST + GraphQL          | 灵活查询 + 标准化端点               |
| 实时通信 | WebSocket (Socket.IO)   | 双向实时通信                        |
| 消息队列 | Bull/Redis              | 异步任务处理                        |
| 容器化   | Docker + Docker Compose | 一致部署环境                        |
| CI/CD    | GitHub Actions          | 与现有流程一致                      |

## 7. 与现有系统的集成策略

### 7.1 渐进式迁移

```
阶段 1: 后端层新建，前端保持现有架构
  ┌─────────────┐     ┌──────────────┐
  │ 前端 (现有)  │────▶│ API 网关 (新) │
  └─────────────┘     └──────┬───────┘
                             │
                    ┌────────┴────────┐
                    │ 后端服务 (NestJS) │
                    └─────────────────┘

阶段 2: 前端逐步对接后端 API
  ┌─────────────┐     ┌──────────────┐
  │ 新前端模块   │────▶│ API 网关     │
  │ 现有前端模块  │────▶│ (Mock 兼容)  │
  └─────────────┘     └──────────────┘

阶段 3: 全面对接，移除 Mock
  ┌─────────────┐     ┌──────────────┐
  │ 全部前端模块  │────▶│ API 网关     │
  └─────────────┘     └──────────────┘
```

### 7.2 前端适配策略

- 新增 `api-service` 层封装所有 API 调用
- 现有 Mock 数据保留为回退方案
- React Query/SWR 管理服务端状态
- 与现有 Zustand 客户端状态共存
